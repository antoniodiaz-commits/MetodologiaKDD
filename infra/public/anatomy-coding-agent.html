01
LLMs, Reasoning Models, and Agents
The three-layer hierarchy behind every coding tool
Core Insight

Much of the recent progress in practical LLM systems is not just about better models, but about how we deploy them. The surrounding system — tool use, context management, and memory — plays as much of a role as the model itself.

When we talk about the “coding capabilities” of LLMs, people often collapse three distinct concepts into one:

LLM

Core next-token prediction model. The raw engine.

Reasoning Model

An LLM trained to spend more inference-time compute on intermediate reasoning, verification, or search.

Agent Harness

A control loop that decides what to inspect, which tools to call, how to update state, and when to stop.

“The LLM is the engine, a reasoning model is a beefed-up engine (more powerful, but more expensive to use), and an agent harness helps us use the model.” — Sebastian Raschka

Key Definitions

TermDefinition
LLMThe raw model — next-token prediction
Reasoning modelAn LLM optimized to output intermediate reasoning traces and self-verify
AgentA loop that uses a model plus tools, memory, and environment feedback
Agent harnessThe software scaffold that manages context, tool use, prompts, state, and control flow
Coding harnessA task-specific harness for software engineering: code context, tools, execution, iterative feedback

A better LLM provides a better foundation for a reasoning model, and a harness gets more out of this reasoning model. But coding work is only partly about next-token generation. A lot of it is about repo navigation, search, function lookup, diff application, test execution, error inspection, and keeping all the relevant information in context.

“Since, in my view, the vanilla versions of LLMs nowadays have very similar capabilities, the harness can often be the distinguishing factor that makes one LLM work better than another.”

02
The Coding Harness
Six components that make agents work

When we say harness, we mean the software layer around the model that assembles prompts, exposes tools, tracks file state, applies edits, runs commands, manages permissions, caches stable prefixes, stores memory, and more.

Today, this layer shapes most of the user experience compared to prompting the model directly. The harness can make both reasoning and non-reasoning models feel much stronger than they do in a plain chat box.

03
Live Repo Context
Workspace summary and stable facts

When a user says “fix the tests” or “implement xyz,” the model should know whether it is inside a Git repo, what branch it is on, which project documents contain instructions, and so on. Those details often change or affect what the correct action is.

“Fix the tests” is not a self-contained instruction. If the agent sees project documentation, it may learn which test command to run. If it knows the repo root and layout, it can look in the right places instead of guessing.

Repo root & layout  •  Git branch & status  •  Instruction files  •  Recent commits
↓
Workspace Summary + User Request = Combined Prompt

Convergence Across Platforms

Every major coding agent collects essentially the same workspace context — but they name the instruction file differently:

Same mechanism, different names

The naming differs, the mechanism is identical: give the agent structured context before it starts working.

Quality matters

The more precise and well-organized these instruction files are, the better the agent performs — regardless of platform.

04
Prompt Shape and Cache Reuse
Stable prefix + dynamic components

Coding sessions are repetitive: the agent rules stay the same, the tool descriptions stay the same, and the workspace summary stays mostly the same. The main changes are the latest user request, the recent transcript, and short-term memory.

“Smart” runtimes don’t rebuild everything as one giant undifferentiated prompt on every turn. They separate what changes from what doesn’t.

Stable Prefix (cached)

  • General instructions & system prompt
  • Tool descriptions (name, schema, docs)
  • Workspace summary (repo layout, git, docs)

Dynamic Components (rebuilt each turn)

  • Short-term memory (task state, notes)
  • Recent transcript (compressed)
  • Current user request

Convergence Across Platforms

PlatformCaching StrategyDetails
Claude CodeExplicit prefix cachingSystem prompt + tools + CLAUDE.md (~18K tokens) cached; ~10x cost reduction
Gemini CLIImplicit caching (2.5+)Automatic on Gemini 2.5 models; 90% cost reduction; min 4,096 tokens
Codex CLIResponse cachingSandboxed execution with response-level caching
DevinVectorized snapshotsCodebase state captured for efficient retrieval in long-running sessions

The principle is universal: separate what changes from what doesn’t, and avoid reprocessing stable information.

05
Tool Access and Use
Where it stops being chat and starts being an agent

A plain model can suggest commands in prose, but an LLM in a coding harness should actually execute commands and retrieve results — versus the user calling the command manually and pasting results back.

Instead of letting the model improvise arbitrary syntax, the harness provides a pre-defined list of allowed and named tools with clear inputs and boundaries.

Model emits structured tool call  →  Harness validates  →  User approves  →  Execute in sandbox  →  Clip & feed back to model

When the model asks to do something, the runtime runs programmatic checks: Is this a known tool? Are the arguments valid? Does this need user approval? Is the requested path inside the workspace? Only after those checks pass does anything actually run.

Read, write, search, execute — every platform wraps these with a validation and permission layer.

Instead of asking for approval, Devin runs in a fully isolated cloud VM, so the blast radius of any action is inherently contained.

06
Minimizing Context Bloat
The underrated, boring part of good coding-agent design

“A lot of apparent ‘model quality’ is really context quality.” — Sebastian Raschka

Coding agents are even more susceptible to context bloat than regular LLMs during multi-turn chats, because of repeated file reads, lengthy tool outputs, and logs. If the runtime keeps everything at full fidelity, it runs out of context tokens quickly.

Clipping

Truncate any single item that exceeds the token budget. Prevents verbose outputs from monopolizing the prompt.

Transcript Reduction

Summarize older events aggressively, keep recent events rich. Full session history becomes a smaller promptable summary.

Deduplication

Remove repeated file reads from earlier in the session. The model doesn’t need to see the same content twice.

Convergence Across Platforms

PlatformContext WindowCompaction Approach
Claude Code1M tokensAuto-compaction at ~92% usage; async summarization; file read dedup
Gemini CLI1M+ tokens (native)Largest native window reduces pressure; implicit pruning on model side
Codex CLI200K tokensSandboxed responses; aggressive prompt management
DevinVariableProgressive compression; vectorized codebase snapshots; replay timeline

Even Gemini’s massive native context window doesn’t eliminate the need for compaction — it just raises the ceiling. Every platform needs this.

07
Structured Session Memory
Working memory + full transcript

A coding agent separates state into at least two layers. The previous section covered prompt-time use of history (compression). This section is about the storage-time structure: what the agent keeps as a permanent record.

Working Memory

  • Small, distilled, actively modified
  • Current task description
  • Important files list
  • Recent notes & decisions
  • Purpose: task continuity across turns

Full Transcript

  • Complete record, append-only
  • All user requests
  • All tool calls & outputs
  • All LLM responses
  • Purpose: prompt reconstruction & session resumption

Convergence Across Platforms

All platforms implement dual-layer memory, though they expose it differently:

PlatformInstruction MemorySession MemoryCross-Session
Claude CodeCLAUDE.md (hand-written) + MEMORY.md (auto)JSON transcripts on disk.claude/memory/ with indexed notes
Gemini CLIGEMINI.md (hand-written, @imports)YAML+Markdown session files/memory command to reload
Codex CLIAGENTS.mdJSONL session filesSession resumption
DevinTask description + repo docsReplay timeline + vectorized snapshotsLearns from past trajectories
Universal pattern

Hand-written project knowledge (what the team knows) + auto-generated session state (what the agent learned). Every platform needs both.

08
Delegation With Bounded Subagents
Parallel work without losing control

Once an agent has tools and state, the next useful capability is delegation: parallelizing subtasks via subagents. The main agent may need a side answer — which file defines a symbol, what a config says, why a test is failing — without interrupting its main thread.

The tricky design problem is not just how to spawn a subagent but how to bind one: subagents need enough context to be useful but tight enough boundaries to be safe.

Convergence Across Platforms

PlatformSubagent ModelConstraintsSpecialization
Claude CodeCustom agents in .claude/agents/Tool allowlists/denylists, independent contextUser-defined specialists
Gemini CLI4 built-in + custom in .gemini/agents/Isolated tools, own context, no sub-sub-agentsInvestigator, CLI Help, Browser, Generalist
Codex CLISubagent spawnTask scoping, depth limits, sandbox inheritanceGeneral-purpose
DevinParallel VM instancesEach in isolated cloud sandboxMultiple Devins on independent tasks
Inherit for utility, constrain for safety

Every platform balances context inheritance vs. isolation differently, but none skips this component.

Custom vs. built-in

Claude Code and Gemini let you define custom specialists. Devin takes a different approach: you can spin up multiple independent Devins.

09 — The Convergence Thesis

Reviewing Claude Code, Gemini CLI, Codex, and Devin side by side, one pattern emerges clearly: all coding agents are converging on the same architecture.

All platforms implement the same six components. They differ in UX (terminal vs. cloud VM), permission model (approval gates vs. sandbox isolation), and ecosystem integrations — but the architectural bones are identical.

As Raschka notes: “The vanilla versions of LLMs nowadays have very similar capabilities. The harness can often be the distinguishing factor.”

But if the harnesses are also converging, what actually differentiates outcomes?

  • Not the model — GPT-5.4, Opus 4.6, and Gemini 2.5 have similar raw capabilities
  • Not the harness — all implement the same six components
  • Not the tools — read, write, search, execute are universal primitives
The knowledge you feed the agent.

An agent operating on a vague README and scattered comments will produce generic, context-free output. The same agent, on the same model, operating on structured, governed, traceable knowledge artifacts will produce domain-aligned, auditable, maintainable work.

10
The Knowledge Gap
What no agent platform solves

Every coding agent has a slot for project knowledge — CLAUDE.md, GEMINI.md, AGENTS.md, or the repo’s documentation. But these instruction files are just the tip of the iceberg.

What the platform provides

  • ✓ Instruction file loading
  • ✓ Context window management
  • ✓ Tool execution
  • ✓ Session memory
  • ✓ Prompt caching

What it does NOT provide

  • ✗ What to write in instruction files
  • ✗ How to prioritize specs
  • ✗ Which knowledge is valid
  • ✗ How knowledge evolves
  • ✗ Traceability & governance

Without Structured Knowledge

Agents default to “vibe coding” — fast but unmaintainable, with no traceability or governance. The model generates plausible output, but:

  • It doesn’t know why a decision was made (no ADRs, no design rationale)
  • It doesn’t know what constraints apply (no architectural rules, no domain invariants)
  • It can’t distinguish current truth from outdated documentation
  • It produces work that can’t be audited or traced back to requirements
This isn’t a model problem

It’s a knowledge problem. And it’s platform-agnostic — it affects Claude Code, Gemini, Codex, and Devin equally.

11
From Harness to Knowledge
Knowledge-Driven Development and contextual activation
Core Thesis

Agent output quality depends directly on knowledge input quality. Change the knowledge specs, and agent behavior changes automatically — without prompt engineering, fine-tuning, or platform migration.

Three Axes of Knowledge

Knowledge-Driven Development (KDD) organizes knowledge across three axes, each serving a distinct role in the agent’s context:

Knowledge (persistent)

ARCH, DOM, PROD, FEAT, DOC — what the organization knows. Versioned, governed, survives across projects. The organizational memory.

Governance (bridge)

RFC, ADR, RULE — decisions and constraints that connect knowledge to work. Why we decided this.

Work (ephemeral)

WRK-SPEC, WRK-PLAN, WRK-TASK — what’s changing now. Scoped, disposable, traces back to knowledge.

Three Pillars

PillarWhat It MeansWhy Agents Need It
Spec-DrivenStructure knowledge as versioned, machine-readable artifactsAgents need parseable, unambiguous input — not prose wikis
EvolutiveKnowledge grows with practice; each project feeds learnings backAgents improve automatically as knowledge improves
AgenticAgents operate on structured knowledge to automate low-value tasksKnowledge activation replaces prompt engineering

Contextual Activation

The mechanism that connects structured knowledge to agent work is contextual activation. When a work artifact declares which knowledge specs inform it, a multi-step pipeline injects precisely-scoped organizational knowledge into the agent’s context:

WRK-SPEC declares activates: DOM-RISK-001, ARCH-INT-003, ADR-007

→

Domain rules, arch constraints, decisions, API contracts → Agent prompt

This knowledge bundle flows into the agent’s context window — whether that agent runs on Claude Code, Gemini CLI, Codex, or Devin. The pipeline is platform-agnostic because it operates at the knowledge layer, not the harness layer.

The Complete Picture

Knowledge Layer (KDD)

  • Knowledge Specs (ARCH, DOM, PROD)
  • Governance (ADR, RULE, RFC)
  • Work Artifacts (WRK-SPEC, WRK-PLAN)

Platform-agnostic. Survives model changes.

Activation

  • Contextual activation pipeline
  • Knowledge graph traversal
  • Context budget management

Declare → Expand → Filter → Bundle

Agent Layer (Harness)

  • 6 components (context, cache, tools...)
  • Claude / Gemini / Codex / Devin
  • Model execution & tool orchestration

Interchangeable. The mechanism.

The knowledge layer is the durable asset. Models will change. Platforms will evolve. Token limits will grow. But the organizational knowledge — domain models, architectural decisions, business rules, API contracts — persists across all of them.

Key Takeaways

The agent is the mechanism. The knowledge is the differentiator.
  • All coding agents converge on the same six components. Claude Code, Gemini CLI, Codex, and Devin implement the same architecture with different trade-offs.
  • The harness matters more than the model. A lot of apparent “model quality” is really context quality.
  • But the harness is converging too. If both models and harnesses are commoditizing, the lasting differentiator is the knowledge you feed the system.
  • Structured knowledge is platform-agnostic. Well-governed specs activate correctly on any agent platform. No vendor lock-in at the knowledge layer.
  • Agents are knowledge-activated executors. Change the knowledge specs, and agent behavior changes automatically.
  • The real question is not “which agent?” but “what knowledge?” How you structure, govern, and evolve organizational knowledge determines agent effectiveness more than model choice or harness features.
